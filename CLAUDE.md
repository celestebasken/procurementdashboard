# CLAUDE.md

Project context for Claude Code. Read this before starting any session in this repo.

## What this project is

A single, multi-tab Streamlit dashboard consolidating four previously separate tools for UC dining sustainability procurement:

1. **Campus Roadmap** ‚Äî optimization (min spend / max sustainable spend / threshold-then-maximize) across food categories (grouped per SIMAP-57 ‚Äî beef vs. chicken rather than individual cuts, generalized from an existing R `lpSolve` model that was previously Berkeley-meat-specific). **Critical: "sustainable" in the optimization objective means `products.validated_sustainable_yn` (AASHE STARS for Academic campuses, Practice Greenhealth for Health campuses) ‚Äî never SIMAP-57 category membership.** SIMAP-57 is only the category grouping the optimizer operates over and the basis for reporting GHG-equivalent footprint alongside the results; it plays no role in defining what counts as "sustainable." Auto-generates a PDF report of current purchasing (including both % sustainable-by-AASHE/PGH and GHG-equivalent emissions) plus a target-setting/roadmap builder.
2. **Dining Dashboard** ‚Äî cross-campus search for sustainable products by category/vendor, so chefs can onboard new sustainable items through vendors they already use. Price-free by design. Rebuild of an existing Python Streamlit app (in `legacy/dining_dashboard/`) on the new canonical schema.
3. **Auto-Classifier** ‚Äî campus uploads a purchasing sheet, gets `sustainability_certifications` auto-filled by matching against previously classified products.
4. **Competitive Price Checker** ‚Äî tests whether a hypothetical new item would be cost-competitive enough to enter a campus's optimized purchasing plan.

**Architecture principle:** all four are thin front-ends over one shared canonical dataset and two shared engines (entity-matching, optimization) ‚Äî not four independently-cleaned pipelines. See schema below before writing any ingestion or matching code.

## Database schema (canonical layer ‚Äî the foundation everything else builds on)

SQLite. Core tables:

**`purchases`** ‚Äî one row per unique item per campus per FY (aggregated from raw transaction-level rows, not one row per transaction)
`campus, fiscal_year, product_id, vendor, brand, total_price, total_weight_lbs, weight_source (reported / computed_tier2 / reference_table_tier3 / unresolved), unit_price, purchase_type (service/purchasing), n_transactions_aggregated, source_report_id`
- GHG total is *computed*, not stored as an independent fact: `total_weight_lbs` (‚Üíkg) √ó the product's SIMAP category factor from `simap_taxonomy`. Caching the computed value as a column for query speed is fine; treating it as independently editable is not. **A row with `weight_source = unresolved` has no reliable GHG total** ‚Äî surface that rather than computing against a missing/zero weight.
- `vendor` means **distributor** specifically ‚Äî the high-level entity that delivers to and bills the campus (e.g. "Sysco," "US Foods"). This is the field entity-matching's vendor-equality gate uses, and it's reliably populated across every campus's export.
- `brand` is a distinct, smaller-grain concept (manufacturer/farm/sub-vendor ‚Äî e.g. "Marys," "Diestel," "Glenview Farms") that several campus exports report in a separate column alongside distributor, and that column is frequently blank. **Never use `brand` for the entity-matching vendor gate or as a Distributor fallback** ‚Äî confirmed with project owner after finding a real bug where UCSD_H's ingestion fell back to its `Supplier` column (a `brand`-grain field) whenever `Distributor` was blank, conflating two different concepts. When Phase 2 merges two products, `brand` values from both sides are preserved comma-separated rather than one being dropped.

**`products`** ‚Äî canonical identity table (the institutional memory)
`product_id, canonical_name, simap_category, simap_classification_source, sustainability_certifications (as reported by campus), sustainable_yn, certification_validation_flag, validated_sustainable_yn, first_seen_fy, last_seen_fy`
- `simap_category` is populated by `lib/simap_classification.py` (Phase 3), which tries three tiers in order and records which one won in `simap_classification_source`: `'campus_category'` (highest confidence ‚Äî several campus exports already carry their own category-ish column: `env_sub_category` for UCD_H/UCLA_H/UCSD_H, `Product Category`/`Product Subcategory` for UCD, `Food Type (WRI)`/`(RFC)` for UCSC, looked up against `reference/simap_keyword_dictionary.csv`), `'keyword_match'` (no campus signal, or it didn't hit the dictionary ‚Äî fall back to matching dictionary keywords against `canonical_name` itself), or `'unclassified'` (left `NULL`, not guessed ‚Äî same philosophy as `weight_source = 'unresolved'`). UCB has no category column at all; UCR's only candidate (`Cost Category`) was deliberately excluded from the dictionary since it's too coarse to trust (e.g. "Meats" spans beef/chicken/pork, which have very different GHG factors) ‚Äî both go straight to keyword_match.
- `reference/simap_keyword_dictionary.csv` (456 entries) started as a port of the legacy R script's hand-built `env_sub_category` ‚Üí SIMAP dictionary (`legacy/cleaning_scripts/DataCleaning1.Rmd`, 9 target-name mismatches against `simap_taxonomy.food_category` corrected during the port), but that source dictionary was produce-specific ‚Äî it was extended substantially (meat/deli, dairy, sauces, baked goods, beverage/snack/brand names) across several rounds by auditing real unclassified output plus two manual passes from the project owner spot-checking the review CSV, taking coverage from ~59% (campus-category tier alone) to 90.1% classified overall.
- **No generic "sauce" or "condiment" category exists in `simap_taxonomy`** ‚Äî SIMAP-57 expects a sauce/paste/condiment to be classified by its dominant base ingredient instead (soy sauce ‚Üí Soybeans/Tofu, marinara ‚Üí Tomatoes, miso ‚Üí Soybeans/Tofu, tamarind paste ‚Üí Stimulants & Spices (misc.)), which is why the dictionary has many specific sauce/paste entries rather than one catch-all. Same reasoning applies to bottled water: there's no water category either, so water products (e.g. "Bubly") are correctly left `unclassified` rather than force-mapped to `Liquids` (which is reserved for soda/juice/tea/other genuine beverages) ‚Äî if the project owner wants water fully excluded, that belongs in the Phase 1 non-food filter (`lib/ingestion.py`'s `NON_FOOD_KEYWORDS`), not this dictionary. In `keyword_match()`, the longest matching keyword wins (more specific phrases beat generic single words); ties on length are broken by **earliest position in the text**, not dictionary order ‚Äî real audit example: "CHICKEN BREAST BUFFALO BITES" has "Chicken" and "Buffalo" both length 7, and the first-mentioned word is the primary ingredient (Poultry), not a later flavor/style descriptor (Spices). Multi-word compound keywords (e.g. "Soy Sauce") only match that exact word order ‚Äî many campus exports write "SAUCE, SOY" instead, so where the bare word alone is safe/specific enough (e.g. "Soy"), it's used instead of the compound, since keyword matching doesn't care about word order or position, only presence.
- Unclassified products get exported for review to `data/processed/unclassified_simap_review.csv` ‚Äî a meaningful share of what lands there (~17% in practice) is genuinely non-food that slipped past the Phase 1 keyword filter (paper goods, cleaning chemicals, trash liners, supplements found there), not a classification gap; the rest is composite dishes, generic descriptors ("snack", "entree", "sauce" alone), and abbreviations where no single correct SIMAP category exists without more context. Left `unclassified` rather than guessed, same philosophy as `weight_source = 'unresolved'`.
- `sustainable_yn` is **campus-reported and trusted as source of truth**, not derived from certification matching ‚Äî campuses are usually right about sustainability status even when they name the wrong certification. **Never overwritten**, kept for audit/provenance.
- `NA` means **unknown/not-yet-evaluated**, not "non-food." In the raw historical data, campuses have generally used NA to mean "non-food line" (gloves, mops, etc.) ‚Äî those lines get filtered out entirely during ingestion (see below), which frees NA to mean what it should mean going forward.
- `certification_validation_flag` is a QA signal (does the reported cert appear in `certification_types` for the applicable framework?) ‚Äî a mismatch gets flagged for review, it never silently overwrites the campus's own Y/N.
- `validated_sustainable_yn` is **derived, not campus-reported** ‚Äî the field the optimizer and reporting should read instead of raw `sustainable_yn`. Computed at ingestion time: if `sustainable_yn = 'Y'`, it stays `'Y'` only when either no certification was claimed (nothing to invalidate) or at least one claimed certification actually matches `certification_types` for the applicable framework; otherwise it's downgraded to `'N'`. `sustainable_yn` values of `'N'`/`'NA'` pass through unchanged. Rationale: a campus claiming "sustainable" on the strength of a certification that isn't a recognized AASHE STARS / Practice Greenhealth standard shouldn't count as validated for this project's purposes, even though the raw claim is preserved untouched in `sustainable_yn`.

**`certification_types`** ‚Äî controlled vocabulary, loaded from `reference/certification_types.csv`
`certification_name, abbreviation, frameworks (multi-value: AASHE STARS and/or Practice Greenhealth), qualifier, needs_review`
- `certification_name` is the unique key ‚Äî **not** framework, since most certifications count under both standards.
- `qualifier` is free text, populated only for genuine restrictions (GAP's varying step/level requirements, geography like "US only," product-subtype like "farmed mollusks only"). It is not a food-category partition ‚Äî most food-category groupings in the source vendor guide were organizational, not regulatory.
- Currently 53 rows, all confirmed, none flagged for review.

**`product_aliases`** ‚Äî every raw name ever seen, linked to a `product_id`
`raw_name, campus, product_id, match_confidence, human_confirmed`
- This table is what makes the Auto-Classifier possible with near-zero extra work later, and what the Dining Dashboard rebuild needs. Probably the single highest-leverage table in the project ‚Äî shared by 3 of the 4 components.

**`product_match_candidates`** ‚Äî entity-resolution staging table (`lib/entity_matching.py`), not a source-of-truth table
`candidate_id, campus (legacy, see below), campus_a, campus_b, product_id_a, product_id_b, match_score, status (pending/approved/rejected)`
- Two matching passes, sharing the same merge mechanics (`merge_products()`) and confidence tiers, both computed by `rapidfuzz.token_sort_ratio` on `canonical_name` (via `_score_matrix()`, which first strips a trailing `(SUS)` campus-reporting tag and normalizes curly quotes ‚Äî see `_clean_for_matching()` ‚Äî before scoring; the stored `canonical_name` itself is untouched). Two hard equality gates, both **also required for review-tier candidacy, not just auto-merge** (confirmed with project owner after manually reviewing ~540 real candidates: differing numbers or differing cited origin are "almost never" the same product, so they're excluded from the queue entirely rather than left as noise):
  - **`_numbers_match()`** ‚Äî blocks whenever the two names' numeric tokens differ (pack size, weight, percent, count, size code, catalog/order number, etc.). A high token_sort_ratio score can still hide a changed quantity, e.g. "ONION, RED JUMBO 25-LB" vs "...5-LB" scores 97.6 (a 1-2 character edit barely moves a long string's ratio).
  - **`_origins_match()`** (`ORIGIN_GROUPS`) ‚Äî blocks whenever the two names cite different origin/region markers (Mexico, California, Chile, Peru, Local, Domestic, specific US states, etc.), synonym-grouped so spelling variants of the *same* origin (Mex/Mexico, Chile/Chilean/Chl) don't block a match. Deliberately excludes words that are far more often a food style/variety than literal geography in this data ‚Äî verified against real product names before excluding, not guessed: "Turkey" is essentially always the bird (0 geographic uses found), "French"/"Italian" are recipe descriptors (French vanilla, Italian seasoning), "Canadian" means Canadian bacon (a dish), "Texas" means Texas Toast (a bread style). Bare "Imported" is excluded too ‚Äî not specific enough to compare against another "Imported" (could be two different countries).
  - There's no score that reliably separates a reformatted duplicate from a different product with similar wording (e.g. "salted" vs "unsalted", or a changed origin/quantity, all score in the same 97-98 band as genuine matches) ‚Äî so the **auto-merge threshold is deliberately very high (99.5)**, restricted to near-perfect formatting-only differences (case, whitespace, hyphen-spacing) that also clear both gates above.
  - `>= 99.5` AND both gates pass: merged immediately via `merge_products()`. Nothing here is undo-able, hence the strict threshold ‚Äî **caution:** an audit found 24 products merged before these gates existed/were complete, with real quantity differences baked into summed purchases totals; see "Known data-integrity issue" below.
  - `90‚Äì99.5` AND both gates pass: inserted as `status='pending'` for human review ‚Äî **nothing in `products`/`purchases`/`product_aliases` changes until approved.** The review queue UI (`app/Entity_Match_Review.py`) calls the identical `merge_products()` path on approval, so both tiers share one merge implementation, and lets a reviewer switch between within-campus and cross-campus scope.
  - Below 90, or either gate fails: not recorded.
- `merge_products()` reconciles `sustainable_yn` on merge (prefers a definitive `Y`/`N` over `NA` from either side, rather than blindly keeping `keep_product_id`'s value) ‚Äî audited real "(SUS)"-suffix pairs and found the base/no-suffix row is often `NA` while the "(SUS)" row has a real campus-reported answer; silently keeping only one would discard that. A genuine `Y`-vs-`N` conflict (rare, ~4 of 569 checked pairs) still falls back to `keep_product_id`'s value ‚Äî no principled way to auto-resolve two opposite campus claims. This is a deliberate, narrow exception to "`sustainable_yn` is never overwritten": that principle is about not letting algorithmic *derivation* (cert matching) override a campus's claim, not about combining two campus claims that turned out to describe the same merged product.
- `find_and_merge_within_campus()` matches **within the same campus and the same distributor only** (`purchases.vendor`, never `purchases.brand`) ‚Äî never across distributors, even for an identical product name (e.g. the same brand of crackers sold by two different distributors stays as two separate products for now; the distributor view is load-bearing for other parts of the dashboard). Merging here **sums `purchases` rows** landing on the same campus+FY, since duplicates here are the same campus's own purchase orders.
- `find_and_merge_cross_campus()` matches the same product bought by **different** campuses, blocked by `products.simap_category` instead of distributor (checked against real data: distributor overlap across campuses is too sparse to gate on ‚Äî most campuses use different regional distributors even for the same product). Requires Phase 3 classification to have run first; unclassified products are skipped. Merging here **never sums `purchases`** ‚Äî `campus` differs between the two rows by definition, so `merge_products()`'s per-(campus, fiscal_year) grouping just repoints `product_id` with no summing collision, no code changes needed there. **Auto-merge additionally requires both products share the same `campus_type`** (Academic/Health) ‚Äî crossing that boundary crosses certification frameworks (AASHE STARS vs Practice Greenhealth), which stays a human judgment call in the review queue even at a very high text similarity score.
- `campus_a`/`campus_b` are the source of truth for which campus(es) a candidate spans (equal for within-campus rows, different for cross-campus rows). The original `campus` column is kept only for backward compatibility with data written before cross-campus matching existed ‚Äî added via `lib.db.migrate_schema()` (non-destructive `ALTER TABLE`, backfilled from `campus` for existing rows) since a full `init_db()` rebuild would have wiped an in-progress review session.

**`simap_taxonomy`** ‚Äî loaded from `reference/simap_categories.csv`, all 56 categories including GHG/nitrogen/transport factors, not just category names. **Used for category grouping and GHG-equivalent reporting ‚Äî never as the definition of "sustainable" in any optimization objective.** The optimizer's sustainability objective always means `products.validated_sustainable_yn` (AASHE STARS / Practice Greenhealth), reported alongside ‚Äî not instead of ‚Äî the SIMAP-based emissions figures.

**`food_groups`** ‚Äî loaded from `reference/food_groups.csv` (Phase 4). `simap_category, food_group` ‚Äî groups SIMAP categories into culinary-substitution umbrellas (e.g. beef/poultry/pork/fish/tofu/legumes are all "Protein") purely for the optimizer's reallocation bounds, not a reclassification of SIMAP itself. See "Optimization engine" below for how this is used and why Cheese/Yogurt/Ice cream ended up as their own singleton groups rather than folded into a generic "Dairy" group. A SIMAP category missing from this file falls back to being its own singleton group at query time rather than crashing (`lib.optimization.build_category_baseline`).

**`campuses`** ‚Äî loaded from `reference/campus_types.csv`. Includes all UC campuses, not just the 7 with current purchasing data (schema should tolerate a campus with zero `purchases` rows). `campus_type` determines which certification framework applies during validation (Academic ‚Üí AASHE STARS, Health ‚Üí Practice Greenhealth).

## Ingestion pipeline ‚Äî distinct steps, in order

1. **Header mapping** ‚Äî one config per campus (raw column names vary per campus). Confirm each campus's mapping with the user rather than guessing; a wrong guess here silently corrupts everything downstream.
2. **Non-food line removal** ‚Äî a line that fails to map to any SIMAP-57 category is the signal to route it to review for likely removal (gloves, mops, ketchup dispensers, etc. sometimes appear in POs). Implemented as `lib/ingestion.NON_FOOD_KEYWORDS`/`WATER_KEYWORDS`, conservative and confirmed-safe-only (checked against real data for false positives before adding each one) ‚Äî BAG and CUP were deliberately excluded (real food items key on that packaging language too: candy "in a bag," yogurt "cups"). **Water is treated as non-food** ‚Äî there's no such thing as "sustainable water" for this project's purposes, so plain/sparkling water (a brand that's always water regardless of flavor, like Bubly; generic spring/mineral/bottled water phrasing; known plain-water-only brands) is removed the same as gloves/napkins. Deliberately narrow: bare "Water" is excluded (huge false-positive rate ‚Äî "Tuna in Water," "Ham & Water," "Cake Mix Add Water," "Water Crackers" are real food, not water), and **flavored/ambiguous water (coconut water, Propel, fruit-infused water) is explicitly out of scope** ‚Äî an unresolved judgment call about how much flavoring moves something from "water" to a genuine beverage, not something to guess at silently.
3. **Transaction-level aggregation** ‚Äî group by (campus, fiscal_year, product) and sum price and weight *before* entity resolution, since resolving duplicate rows of the same raw name is wasted work.
4. **Weight resolution (three-tier ‚Äî see dedicated section below)** ‚Äî direct weight column where reported; computed from case size √ó unit descriptor where parseable; reference-item weight table as last resort. Every row gets a `weight_source`; nothing gets silently zeroed or defaulted.
5. **Certification validation** ‚Äî fuzzy-match reported certification names (and abbreviations) against `certification_types`, confirm the applicable framework is in `frameworks`. Flag mismatches via `certification_validation_flag`; `sustainable_yn` itself is never overwritten, but a `'Y'` claim resting on a non-validating certification is downgraded in the derived `validated_sustainable_yn` field (see `products` schema above) ‚Äî that's the field downstream consumers (optimizer, reporting) should read.

Python, Streamlit, SQLite, `rapidfuzz` (entity matching), `PuLP` (optimization, porting the R `lpSolve` model), `WeasyPrint` (PDF report generation).

## Weight resolution (`lib/weight_lookup/`) ‚Äî parsing and inference, not lookup against complete data

This is not a lookup against complete distributor-provided weight files ‚Äî no more external weight data is coming in. Instead, weight has to be **derived per line item**, using whatever partial information each distributor happens to provide, in three tiers:

**Tier 1 ‚Äî direct weight column.** Some distributors report total lbs directly. Straightforward; use as-is.

**Tier 2 ‚Äî computed from case size √ó unit descriptor.** Many distributors give something like case count plus a cut/unit size embedded in the item's text description (e.g. a meat product listing "cases" purchased, with the cut size only mentioned in the item title/description, not a structured column). Resolving this requires: (a) extracting the relevant quantity from the item description ‚Äî this varies by distributor and likely needs a mix of regex and case-by-case judgment, not one universal parser, and (b) multiplying case count √ó extracted unit size. **This logic will differ per distributor** (~15+ distributors across 7 campuses), since each formats their reports differently ‚Äî expect per-distributor extraction rules, not one shared parser that works everywhere.

**Tier 3 ‚Äî no size/weight info at all; fall back to a reference-item weight table.** Some distributors give no usable size info whatsoever (e.g. "100 orders of a dozen bagels," with no weight anywhere) ‚Äî the fallback is a lookup by product identity against a maintained reference table of typical per-unit weights (`reference/weight_dictionaries/` currently holds early examples of exactly this, e.g. "1 bagel = X lbs"). This table is populated by AI + human judgment together (estimating a reasonable average weight for a known food item, then a human confirming or correcting it), not derived from campus data at all ‚Äî it's closer to `certification_types` or `simap_taxonomy` in character (a maintained reference table) than to a per-distributor parser.

**This is inherently incremental, not a one-time task.** Coverage will be built up distributor-by-distributor and item-by-item, mostly during and after Phase 1, using Claude Code as the primary tool for both the extraction-logic work (Tier 2) and the estimation work (Tier 3, with human review). Don't try to achieve full coverage before moving past Phase 1 ‚Äî instead, every resolved weight should carry a **`weight_source`** value (`reported` / `computed_tier2` / `reference_table_tier3` / `unresolved`) so partial coverage is always visible and traceable, not silently treated as equivalent to a directly-reported number. Tier 3 estimates in particular should carry a `human_confirmed` flag, since an AI-estimated average bagel weight is meaningfully less certain than a distributor-reported case weight, and that distinction matters for anyone auditing the optimizer's results later.

**Phase 1 explicitly does not need to solve Tier 3.** Phase 1's ingestion pass should handle Tier 1 (direct weight columns) and attempt Tier 2 (case-size √ó unit-descriptor parsing) where mechanically straightforward, and simply mark anything else `weight_source = unresolved`. Building out the reference-item estimation table (Tier 3) ‚Äî the "AI + human together" work of estimating and confirming per-unit weights for items with no size info at all ‚Äî is a separate, later, ongoing effort. Don't let Phase 1 scope creep into trying to resolve every unweighted item; an honest `unresolved` flag is a complete and correct Phase 1 outcome for those rows.

Practically, this means `purchases.total_weight_lbs` needs a companion `weight_source` column, and the Tier 3 reference table needs the same kind of `human_confirmed` field the entity-matching system already uses in `product_aliases` ‚Äî the two problems (matching a raw name to a known identity, and trusting an inferred value) are structurally similar.

## Optimization engine (`lib/optimization.py`, Phase 4) and its scenarios (Phases 5 & 8)

`build_category_baseline(conn, campus, fiscal_year)` returns one row per SIMAP category (baseline spend/weight, split into sustainable/conventional sub-splits with their own $/lb, plus `food_group` and a GHG factor). Every scenario solver builds a PuLP LP over these rows and returns an `OptimizationResult` (`category_results`, `totals`, `assumptions`).

**Reallocation is bounded three ways, nested** (tightened twice from an original ¬±50%, based on real project-owner feedback that looser bounds produced implausibly large single-category swings):
1. **Global** ‚Äî total food weight across every optimized category is held EXACTLY fixed (the analog of the legacy R model's fixed "meals served").
2. **Food group** (`food_groups` table) ‚Äî each group's total weight may move ¬±15% from its own baseline. This is the guardrail against, e.g., cutting beef and "offsetting" it with unrelated apples ‚Äî substitution is only possible within a group of culinarily-reasonable alternatives.
3. **Individual SIMAP category** ‚Äî each category's own weight may also move ¬±15% from baseline.

**`reference/food_groups.csv`** groups the 55 SIMAP categories into umbrellas: a combined "Protein" group (beef, poultry, pork, lamb, fish, crustaceans, mollusks, tofu, legumes, beans and pulses ‚Äî 10 categories, all animal + plant protein together), "Dairy & Milk Alternatives" (milk/butter/cream/almond/oat/rice/soy milk), "Vegetables", "Fruits", "Grains & Starches", "Oils & Fats" ‚Äî plus a long tail of **singleton groups** (Eggs, Wine Grapes, Tree nuts, Sugars, Cocoa, Stimulants, Liquids, and ‚Äî per project-owner direction ‚Äî **Peanuts/groundnuts** (pulled out of Protein into its own singleton) and **Cheese, Yogurt, and Ice cream** (each split out of a generic "Dairy" bucket into their own singleton, since the eventual goal is comparing them against non-dairy alternatives like dairy-free ice cream, not against milk).

**Price-outlier guard** (`_flag_price_outliers`, inside `build_category_baseline`) ‚Äî added after a real bug was traced to a single corrupted purchases row (UC Berkeley's raw export literally contained `"1,425,014.256 LB"` for a $244.84 case of frozen broccoli bowls, silently driving that category's price toward $0/lb). Flags a row as untrustworthy-for-weight-math (folded into the same bucket as `weight_source = 'unresolved'` ‚Äî spend still counts, weight doesn't) when its implied $/lb is a per-category log-scale MAD outlier at `z > 7.5`. That threshold was calibrated against real data, not guessed: 6 confirmed-corrupted rows all score `z > 11`, while legitimate wide-but-real price spread within one category (e.g. raw bulk potatoes at $0.33/lb next to $4-100+/lb processed chips inside "Potatoes") tops out at `z = 1.89` ‚Äî a lower threshold like the standard 3.5 boxplot rule flagged dozens of these legitimate rows as false positives.

**Three scenarios**, all sharing the same bound structure:
- `solve_min_spend_keep_sustainability` ‚Äî minimize cost, subject to sustainable spend staying at/above baseline.
- `solve_max_sustainable_keep_cost` ‚Äî maximize sustainable spend, subject to cost staying at/below baseline.
- `solve_threshold_third_of_scenario1` ‚Äî runs scenario 1 first to find its achievable cost cut, then locks in only 1/3 of that cut and maximizes sustainable spend under the smaller target (project owner's exact spec, e.g. scenario 1 finds 9% possible ‚Üí this commits to 3%).

`identify_category_movers(category_results)` ranks which categories shifted most toward/away from sustainable spend, filtered to exclude categories with no real baseline choice (all-sustainable or all-conventional data) and below a materiality spend threshold. Its `price_ratio_sus_to_conv` field is the causal explanation for a shift (‚â§1 = a "free win," cost-neutral-or-better; >1 = the scenario's target, not price, is driving it) ‚Äî this is the basis for the Roadmap page's "Key takeaways" panel, in plain language for a non-technical stakeholder audience (chefs and business staff, not just analysts).

**Phase 8 (`HypotheticalItem` + `solve_hypothetical_item_check`)** mirrors the legacy R "Hypothetical Proteins" tab exactly rather than doing a simple $/lb comparison: it injects a proposed new item as a genuine third sourcing option for one existing SIMAP category (its own price, an optional supply cap defaulting to unlimited, a sustainability claim, zero baseline use) into the same real LP, and reports whether ‚Äî and how much ‚Äî the solver actually chose to use it. A great price can still lose if the category/food-group band is already maxed out; a middling price can still win if the scenario's sustainability target needs the extra spend. Implemented as backward-compatible optional parameters threaded through the existing internals (`_add_common_constraints`, `_cost_expr`, `_sus_spend_expr`, `_extract_results` all accept an optional `hyp` tuple, no-op when `None`) rather than a parallel code path, verified by running the full pre-existing test suite unchanged before adding new tests.
